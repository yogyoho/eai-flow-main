# 面向交通工程设计审查的多智能体协同审核平台研究与应用

> 课题设计文档 | 2026-06-10

## 1. 课题背景与问题定义

### 1.1 背景

交通规划设计院在管理职能中承担着对下属设计单位提交的工程报告审核、施工单位验收等关键工作。当前审核工作面临以下系统性痛点：

- **审核效率低**：报告数量大、专业性强，审核人员有限，审查周期长、积压严重
- **审核质量不稳定**：不同审核人员的标准和深度不一致，容易漏审或误判
- **知识传承困难**：老专家退休后，审核经验和规范理解难以传递给新人
- **验收规范化不足**：现场验收缺乏标准化流程和数据支撑，主观性强

### 1.2 问题本质

交通工程设计审查是一个**多专业并行、多规范约束、高经验依赖**的复杂认知任务，天然适合用多智能体协同架构来辅助。

### 1.3 课题定位

横向课题 / 企业委托项目，偏工程化应用和落地验证。委托方为类似交通规划设计院的管理型机构，审核的工程报告覆盖综合类型（路线、桥隧、交安、造价等）。

## 2. 课题名称与研究目标

### 2.1 课题名称

**面向交通工程设计审查的多智能体协同审核平台研究与应用**

### 2.2 研究目标

1. 构建基于多智能体协同架构的交通工程设计审查辅助平台，实现对公路工程、桥梁隧道、交通安全设施、工程造价等多专业设计文件的自动化分拆与并行审查
2. 建立可扩展的审核知识注入机制，将公路工程技术标准、行业设计规范、审核要点等专业知识结构化地嵌入智能体，形成"规范即技能"的知识管理体系
3. 通过人机协同审核流程，在保障审核质量的前提下提升审查效率，沉淀审核经验形成可复用的知识资产

### 2.3 预期成果指标

| 指标 | 目标值 |
|------|--------|
| 单份报告初审时间 | 从人工 3-5 个工作日缩短至 4-8 小时（含人工复核） |
| 规范条文自动匹配准确率 | ≥ 85%（Top-3 召回） |
| 常见问题检出率 | ≥ 90%（基于历史审核意见验证集） |
| 审核意见采纳率 | ≥ 70%（审核人员认为建议有价值） |

| 工作流引擎 | 审核流程管控 | 复用现有工作流引擎，定义审核-复核-确认三阶段流程 |

## 4. Sub-Agent 详细设计

### 4.1 统一输出格式

所有专业 Sub-Agent 输出统一的结构化审核意见 JSON，便于 Lead Agent 汇总和前端展示：

```json
{
  "agent_id": "route-review-agent",
  "agent_role": "路线设计审核",
  "report_sections": ["第3章 路线设计", "附图A-01~A-15"],
  "findings": [
    {
      "severity": "major",
      "category": "规范符合性",
      "location": "3.2.1节 平面线形",
      "description": "JD3圆曲线半径350m，不满足JTG D20-2017表6.2.1一般值要求（400m）",
      "regulation_ref": "JTG D20-2017 第6.2.1条",
      "suggestion": "建议调整JD3半径至不小于400m，或补充论证说明采用最小值的依据"
    }
  ],
  "summary": "路线总体方案合理，共发现2个重大问题、3个一般问题，建议修改后重新提交",
  "confidence": 0.85
}
```

字段说明：
- `severity`：`major`（重大问题，必须修改）/ `minor`（一般问题，建议修改）/ `info`（提示性信息，供参考）
- `category`：问题分类，如"规范符合性"、"设计合理性"、"文件完整性"等
- `location`：问题在报告中的具体位置（章节、图号、表号）
- `description`：问题的具体描述
- `regulation_ref`：对应规范条文编号
- `suggestion`：修改建议
- `confidence`：Agent 对此条意见的置信度（0-1），低于阈值的标记为"待人工确认"

### 4.2 Agent 1：路线设计审核 Agent

| 维度 | 设计 |
|------|------|
| **角色定位** | 公路路线设计审核专家，精通平面线形、纵断面、横断面设计规范 |
| **审查范围** | 路线方案比选、平面设计、纵断面设计、横断面布置、路基设计 |
| **核心 Skills** | `route-design-review/SKILL.md` — 内嵌 JTG D20-2017、JTG B01-2014 中路线相关条文审查清单 |
| **专属 Tools** | ① 线形指标验算：读取设计参数，验算圆曲线半径、缓和曲线长度、停车视距等是否满足规范值 ② 超高加宽计算：根据设计速度和曲线半径，复核超高/加宽过渡段是否正确 ③ 土石方估算比对：对路线方案进行简易土石方估算，与报告数据交叉验证 |
| **触发条件** | Lead Agent 识别到报告中包含"路线设计"、"平面线形"、"纵断面"等关键词或章节 |

### 4.3 Agent 2：桥隧工程审核 Agent

| 维度 | 设计 |
|------|------|
| **角色定位** | 桥梁与隧道工程审核专家，精通结构设计规范、施工方案评审 |
| **审查范围** | 桥梁总体布置、上下部结构设计、隧道衬砌与防排水、地质适应性 |

| **触发条件** | 报告中包含"桥梁设计"、"隧道工程"、"结构计算"等章节 |

| 维度 | 设计 |

### 4.6 Agent 5：综合合规性审核 Agent

| 维度 | 设计 |
|------|------|
| **角色定位** | 综合合规审核专家，负责检查报告整体合规性、完整性和格式规范性 |
| **审查范围** | 报告编制依据完整性、章节结构合规性、强制性条文执行情况、前后一致性 |
| **核心 Skills** | `compliance-review/SKILL.md` — 内嵌各类设计文件编制办法、强制性条文汇总 |
| **专属 Tools** | ① 章节完整性检查：对照编制办法逐项核对报告章节是否齐全 ② 强制性条文逐条核对：检索报告中是否逐条回应了相关强制性标准 ③ 前后数据一致性：检查报告中不同章节引用的设计参数是否一致 |
| **触发条件** | 所有报告必触发，作为基础审查项 |

### 4.7 调度与协同机制

```
报告上传
    │
    ▼
┌─────────────────────────┐
│  Lead Agent：报告解析     │
│  · 提取目录结构和章节标题  │
│  · 识别涉及的专业维度     │
│  · 制定审查任务分配方案    │
└─────────┬───────────────┘
          │
          │  综合合规 Agent 始终启动
          │  其他 Agent 按需动态启动
          │
    ┌─────┼──────┬──────┬──────┐
    ▼     ▼      ▼      ▼      ▼     ← 全部并行
  合规   路线   桥隧   交安   造价
  Agent  Agent  Agent  Agent  Agent
    │     │      │      │      │
    └─────┴──────┴──────┴──────┘
          │
          ▼
┌─────────────────────────┐
│  Lead Agent：意见汇总     │
│  · 合并各专业审核意见     │
│  · 冲突检测              │
│  · 按严重程度排序        │
│  · 生成结构化初审报告    │
└─────────────────────────┘
          │
          ▼
   人工复核 → 确认/修改 → 最终审核意见
```

核心设计要点：

1. **按需动态调度** — 不是每份报告都启动全部 5 个 Agent，而是根据报告内容智能识别需要哪些专业审查
2. **并行执行** — 各 Agent 独立运行、互不干扰，通过结构化 JSON Schema 对齐输出格式
3. **冲突检测** — Lead Agent 汇总时交叉比对各专业意见，发现矛盾点并标注供人工决策
4. **渐进式知识积累** — 每次审核的经验沉淀到 Memory，后续同类项目的审查准确率逐步提升

## 5. 技术路线与实施计划

### 5.1 第一阶段：知识基座构建（3-4 个月）

核心工作：

- 将各规范拆解为审查清单：每条规范 → 检查项 + 判定逻辑 + 常见问题模式
- 建立历史审核意见的标注体系，作为准确率验证基准
- 初步搭建 5 个专业 Skill 包

知识来源与输出：

| 输入规范 | 输出 Skill |
|---------|-----------|
| 公路工程技术标准 JTG B01-2014 | route-design-review/SKILL.md |
| 公路路线设计规范 JTG D20-2017 | route-design-review/SKILL.md |
| 公路桥涵设计通用规范 JTG D60 | bridge-tunnel-review/SKILL.md |
| 公路隧道设计规范 JTG D70/2 | bridge-tunnel-review/SKILL.md |
| 公路交通安全设施设计规范 JTG D81-2017 | safety-facility-review/SKILL.md |
| 公路工程概预算编制办法 JTG/T B06-01 | cost-review/SKILL.md |
| 各类设计文件编制办法 | compliance-review/SKILL.md |

同时利用历史审核意见 200+ 份进行 Memory 初始化，人工标注问题类型和严重程度，构建问题模式库。

### 5.2 第二阶段：多智能体协同审查引擎开发（4-5 个月）

- 基于 DeerFlow Lead Agent 定制审核调度 Prompt，实现报告结构解析和任务分发
- 开发各专业 Sub-Agent 的专属 Tools（线形验算、结构校核、造价比对等）
- 实现结构化输出 Schema，确保各 Agent 意见格式统一
- 开发冲突检测逻辑：跨专业意见交叉比对
- 对接前端审核工作台：报告上传 → 审核看板 → 意见确认界面

### 5.3 第三阶段：试点验证与优化（3-4 个月）

- 选取委托方 20-30 份已完成审核的历史报告作为回溯验证集
- 人工标注 vs 智能审查结果盲比对照，量化准确率和漏审率
- 在委托方实际工作流中开展双轨并行试运行：智能初审 + 人工同步审核
- 收集反馈，迭代优化 Prompt、Skill、Tool

## 6. 关键技术创新点

### 6.1 创新点 1：规范即技能（Regulation-as-Skill）的知识注入模式

将每类规范封装为 DeerFlow Skill，而非传统的 RAG 检索或全量 Prompt 注入：

```
route-design-review/SKILL.md
├── 角色定义（你是路线审核专家，审查依据为...）
├── 审查清单（按规范条文逐条展开的检查项）
├── 常见问题模式（从历史审核中提炼的高频问题）
├── 判定规则（什么情况算不合规，什么情况需要补充论证）
└── 参考资源（规范原文索引、示例报告）
```

优势：可独立更新、可组合复用、按需加载不浪费上下文。规范更新时只需替换对应 Skill，不影响其他 Agent。

### 6.2 创新点 2：多智能体分角色并行审查 + 跨专业冲突检测

单个 LLM 做全专业审核容易遗漏和顾此失彼。本课题通过多个独立 Sub-Agent 并行审查：

- 每个 Agent 只聚焦一个专业领域，上下文干净、审查深入
- Lead Agent 汇总时执行跨专业冲突检测，发现人容易忽略的专业间矛盾

### 6.3 创新点 3：审核经验的持续沉淀与自适应优化

通过 DeerFlow 的 Memory 机制：

- 每次审核结论沉淀为结构化案例（问题 → 规范依据 → 审核意见 → 最终处理结果）
- 后续同类项目自动检索相似历史案例，提升审查一致性
- 审核人员的修正反馈反哺系统，形成越用越准的正循环

10-13 个月。
