# 煤炭环评报告编写技能设计方案

> **日期**: 2026-06-12
> **状态**: 待实施
> **范围**: 煤炭行业环境影响评价报告（矿区总体规划环评）智能编写技能

---

## 1. 概述

### 1.1 背景

煤炭行业环境影响评价报告书（环评报告）是大型技术文档，通常 400-700+ 页，涵盖自然环境现状调查、环境影响预测与评价、环保措施论证、合规性分析等专业内容。编写过程涉及多学科知识（大气、水、生态、噪声、地质等）、大量专业计算、法规标准引用和表格图表处理。

当前系统已有：
- **知识工厂**：报告模板抽取与管理（`knowledge-factory` MCP）
- **项目工作流**：章节级创建、编辑、审批（`project` MCP）
- **RAGFlow**：知识库向量检索
- **消防报告技能 v2**：成熟的模板驱动报告生成模式

本技能在此基础上，为煤炭环评报告提供智能编写能力。

### 1.2 目标

1. 在项目工作流中按章节逐步生成煤炭环评报告内容
2. 从知识工厂模板获取章节结构和写作规范
3. 通过多知识库 RAG 检索获取方法论参考
4. 支持三种计算模式（内置脚本 / 占位符 / 外部 API）
5. 确保仿写而非照抄（实体隔离机制）
6. 双重合规检查（模板规则 + 法规知识库）

### 1.3 约束

- **不新增 MCP 服务器** — 复用现有 `knowledge-factory`、`project` MCP 和知识库 REST API
- **不修改 harness 核心代码** — 仅新增技能目录和计算脚本
- **单次生成 = 一个章节** — 700+ 页报告无法一次完成
- **实体数据由用户提供** — 绝不从样例报告复制实体数据

## 3. 仿写约束机制（四重防护）

### 3.1 问题定义

样例报告内容可分为两层：

| 层次 | 内容示例 | 正确处理 |
|------|----------|----------|
| **方法论层**（应仿写） | 分析思路、论证逻辑、章节结构、评价方法、技术路线、表述范式 | ✅ 学习仿照 |
| **实体数据层**（严禁照抄） | 项目名称、地理位置、产能、监测数据、敏感目标、生态特征、地质参数 | ❌ 必须替换 |

### 3.2 防护一：项目实体卡片

在步骤1中建立，贯穿所有章节生成过程：

```
## 项目实体卡片

### 项目基础
- 项目名称: [用户填写]
- 矿区名称: [用户填写]
- 地理位置: [省/市/县/经纬度]
- 评价范围: [面积 km²]
- 建设单位: [用户填写]
- 评价单位: [用户填写]

### 矿区产能
- 矿井名称与产能: [矿井A: X Mt/a, 矿井B: Y Mt/a, ...]
- 总规模: [Mt/a]
- 煤种: [用户填写]

### 敏感目标
- 自然保护区: [名称, 距离, 级别]（无则填"无"）
- 水源地: [名称, 距离]
- 文物古迹: [名称, 保护级别]
- 居民点: [主要村镇名称及距离]

### 环境特征
- 地形地貌: [如: 黄土丘陵/沙漠/平原]
- 气候类型: [如: 温带大陆性干旱气候]
- 主要水系: [名称]
- 土壤类型: [描述]
- 植被类型: [描述]

### 引用标准
- 环评导则: HJ130-2019, HJ463-2009
- [按项目实际需要的标准列表]
```

**存储方式**：项目实体卡片以 JSON 格式存入项目元数据（通过 `project` MCP），支持跨会话恢复。

**核心规则**：生成任何章节时，所有实体数据必须从卡片取值。卡片中标注 `[待补充]` 的字段，报告中同样用 `[待补充: 需填入XX信息]` 占位，绝不从样例报告复制实体数据。

### 3.3 防护二：RAG 检索策略

区分两种检索意图：

| 检索意图 | 查询方式示例 | 用途 |
|----------|--------------|------|
| **方法论检索** | "沉陷预测的评估方法有哪些""大气环境影响预测的技术路线" | 获取分析框架和论证逻辑 |
| **参考格式检索** | "水环境质量现状评价表格格式""环境空气监测数据呈现方式" | 获取表格/图表的结构范式 |

**禁止的查询模式**：
- 禁止用样例项目实体做查询词（如"横城矿区空气质量"）
- 检索结果中的具体数值仅作为方法论参考（"该章节通常包含哪些类型的监测数据"），不作为内容来源

### 3.4 防护三：实体泄露检测

在步骤5合规检查前执行。维护 `references/sample_entities.md` 样例实体黑名单：

```
## 样例报告实体黑名单

### 项目名称
- 横城矿区、横城矿区总体规划

### 矿井名称
- 马莲台煤矿、任家庄煤矿、红石湾煤矿、丁家梁煤矿

### 地理位置
- 灵武、宁东、甜水河、鸭子荡水库

### 敏感目标
- 白芨滩国家级自然保护区、明长城

### 企业名称
- 宝丰能源、神华宁煤、宁夏煤业

### 具体数值（需根据实际项目替换）
- 8.3Mt/a、6.0Mt/a、2.4Mt/a、58km²
```

**检测流程**：对每个生成的章节，扫描黑名单实体 → 发现泄露时标记为 🔴 → 自动替换为项目实体卡片中的对应值 → 无法对应时替换为 `[待补充]`。

### 3.5 防护四：SKILL.md 显式规则

在 SKILL.md 顶部关键规则区声明仿写约束（详见第4.1节）。

## 4. 技能工作流

### 4.1 SKILL.md 关键规则

```markdown
## ⛔ 关键规则

### 关于输出方式

1. 本技能在项目工作流中按章节生成**结构化 Markdown 文本**，通过 `project` MCP 的 `write_chapter` 写入。
2. 生成后用户在项目工作流中审阅、编辑、提交审批。
3. 不直接生成 .docx 文件。

### 关于模板获取

4. 优先调用 `kf_resolve_template` 获取知识工厂模板元数据。
5. 仅当 MCP 工具返回 `found=false` 或调用失败时，才回退到读取 `references/` 下的参考文档。

### 关于仿写约束（极其重要）

6. 样例报告是**方法论和结构参考**，不是内容来源。
7. **禁止原样复制**样例报告中的任何段落、表格数据或数值。
8. 从样例报告中只学习：论证逻辑、专业术语和表述方式、表格结构范式、评价方法选用。
9. **必须替换**样例中的所有实体数据（项目名/地名/产能/数据/敏感目标）。
10. 缺少数据时用 `[待补充: 需提供XX数据]` 占位，**绝不**用样例数据填充。
11. 生成后自动进行实体泄露检测。

### 关于执行顺序

12. 每次对话轮次生成**一个章节**（或一个二级子章节）。
13. 通过 `list_chapters` 查找下一个 `status=pending` 的章节继续。
14. 章节生成顺序遵循批次依赖关系（批次1→2→3→4→5）。
```

### 4.2 初始化流程（首次使用时执行）

**步骤1：了解需求 + 建立项目实体卡片**

与用户交互，确定以下信息：

必须确认（缺少则追问）：
- 项目名称和矿区名称
- 报告类型：总体规划环评 / 建设项目环评 / 规划环评跟踪
- 评价范围（km²）

尽量收集（缺少时标注 `[待补充]`）：
- 矿区现有矿井名称和产能
- 地理位置（省/市/县）
- 用户选择的计算模式：内置脚本 / 占位符 / 外部 API
- 是否有已批复的环评或验收文件
- 设计/评价单位名称

将收集的信息组织为**项目实体卡片**，存入项目元数据。

**步骤2：解析报告模板**

```
kf_resolve_template(
    domain_keywords=["煤炭环评报告", "矿区总体规划环评", "环境影响评价报告书"],
    industry="煤炭",
    report_type="环境影响评价",
    min_completeness_score=60
)
```

- 成功（`found=true`）：使用模板的 `sections` 作为章节结构，每章有 `generation_hint`、`compliance_rules`、`content_contract`
- 失败：回退到 `references/report_structure.md`

**步骤3：加载补充知识**

始终读取以下参考文档：
- `references/terminology.md` — 环评专业术语
- `references/content_guidelines.md` — 各章节编写规范
- `references/sample_entities.md` — 样例实体黑名单

### 4.3 章节生成循环（每轮对话执行一次）

```
单章节生成微流程：

1. list_chapters(project_id)
   → 找到第一个 status=pending 的章节

2. get_chapter_spec(chapter_id)
   → 获取 generation_hint / compliance_rules / content_contract / 邻章上下文

3. 加载项目实体卡片
   → 确保所有实体数据从卡片取值

4. RAG 方法论检索（按章节类型选择知识库）
   → 样例库: 该章的论证思路和结构范式
   → 导则库: 该章适用的技术导则要求
   → 法规库: 该章相关的标准条款

5. 如涉及计算，按用户选择的模式处理
   → 内置脚本: python scripts/calc/calc_xxx.py --params '...'
   → 占位符: [待补充: 需使用XX模型，输入参数为...]
   → 外部API: 调用配置的API端点

6. 生成章节内容（约2000-8000字）
   → 遵循 generation_hint
   → 覆盖 content_contract.key_elements
   → 实体数据从卡片取值

7. 实体泄露检测
   → 扫描 sample_entities.md 黑名单
   → 自动替换或标记

8. write_chapter(chapter_id, content, status="draft")

9. 输出摘要给用户
   → 章节标题 + 字数 + 关键结论
   → 实体泄露检查结果
   → 下一步建议
```

### 4.5 收尾流程（全部章节完成后）

**步骤5：三重检查**

5a. 实体泄露检测：扫描全篇，确认无样例实体残留

5b. 模板合规规则校验：
- 对每个章节，逐条检查 `compliance_rules` 中的标准条款
- 输出：`✅ 通过` / `⚠️ 需复核` + 条款说明

5c. 法规知识库 RAG 对照：
- 针对报告内容，从法规标准库检索相关 GB/HJ 标准条款
- 比对报告结论与标准要求的一致性
- 输出：合规检查摘要表

**步骤6：质量报告**

- 各章节字数统计 vs `word_count_target`
- `content_contract.key_elements` 覆盖率
- 合规检查通过率
- 计算结果占位符清单（如有）
- 建议人工复核的重点章节

## 5. 计算工具

### 5.1 三种计算模式

**模式一：内置脚本计算**
- 调用 `scripts/calc/` 下的 Python 脚本
- 输入：项目参数（从实体卡片获取）
- 输出：计算结果 JSON
- 适用场景：简化估算（AERSCREEN 级别）、噪声衰减、水量平衡

**模式二：占位符**
- 生成 `[待补充: 需使用 XX 模型进行计算，输入参数为...]`
- 附带计算方法说明和所需参数清单
- 适用场景：初稿阶段或用户自行计算

**模式三：外部 API**
- 调用预配置的计算服务 API
- 需在 `extensions_config.json` 中配置 API 端点
- 适用场景：已部署专业计算软件（AERMOD/CALPUFF）

### 5.2 内置计算脚本清单

| 脚本 | 功能 | 输入参数 | 输出 |
|------|------|----------|------|
| `calc_subsidence.py` | 概率积分法沉陷预测 | 开采厚度、倾角、深厚比、煤层参数 | 下沉值、水平移动、变形值 |
| `calc_noise.py` | 噪声衰减计算 | 源强、距离、衰减系数、屏障 | 各距离点噪声级 |
| `calc_water_balance.py` | 矿区水量平衡 | 用水量、供水量、排水量 | 供需平衡表 |
| `calc_air_screen.py` | AERSCREEN 简化大气估算 | 排放速率、烟囱参数、气象 | 最大落地浓度、距离 |
| `calc_capacity.py` | 大气/水环境容量估算 | 环境质量目标、现状浓度、排放量 | 剩余容量 |

统一调用接口：
```bash
python scripts/calc/calc_xxx.py --params '{"param1": value1}' --output json
```

## 10. 会话连续性

用户可能跨多个会话完成整篇报告。

**同一会话内**：agent 在内存中维护项目实体卡片和进度，每完成一章输出进度摘要。

**跨会话恢复**：
1. 新会话开始时，调用 `list_chapters()` 查看各章状态
2. 找到第一个 `status=pending` 的章节继续
3. 通过 `get_project()` 恢复项目实体卡片
4. 如果项目元数据中无实体卡片，提示用户重新提供
