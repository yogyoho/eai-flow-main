## 1. 背景与目标

煤炭领域招投标和供应链互联网平台积累了海量数据，包括供应商、招标、采购、财务、供应链等数据。本设计定义如何将这些数据构建为高质量数据集，并通过 MCP 服务订阅、数据集售卖、AI 训练平台三种模式实现数据资产变现。

### 现状

- **数据存储**: 混合存储（结构化数据库 + 非结构化文档）
- **数据量级**: 100万-1000万条记录 + 大量 PDF/Word 文档
- **变现方向**: AI 数据服务（MCP/API）为核心，数据集和训练平台为辅
- **策略**: 渐进式——先嵌入 EAI Flow 平台验证，后独立扩展

### 2.3 数据加工六阶段流水线

结构化数据从原始状态到高质量数据集，经过六阶段流水线处理：

```
数据探查 → 数据清洗 → 数据标准化 → 数据加工富化 → 质量治理 → 数据标注
   ↓           ↓           ↓            ↓            ↓          ↓
 知己知彼    去脏去重     统一语言      增值赋能      持续保障    AI就绪
```

#### 阶段 1：数据探查（Data Profiling）

先摸清数据"长什么样"，为后续清洗提供依据。

| 探查维度 | 内容 | 煤炭行业示例 |
|---------|------|------------|
| 数据量 | 每张表/字段的记录数、增长速率 | 供应商表 50,000 条，日增约 20 条 |
| 完整性 | 各字段空值率、必填字段缺失率 | "资质等级"空值率 35%，需补全 |
| 唯一性 | 主键冲突、近似重复记录 | 同一供应商以不同名称出现 3-5 次 |
| 分布 | 数值分布、枚举值频率、时间跨度 | 中标价集中在 ¥400-700/吨，异常值 ¥50 |
| 关联性 | 外键完整性、跨表一致性 | 招投标表引用的供应商 ID 15%在供应商表中不存在 |
| 格式 | 日期格式、编码格式、度量单位 | 同字段出现多种日期格式 |

技术实现：
```
tools/
├── profiler.py           # 自动化探查引擎
│   ├── scan_table()      # 扫描单表：类型推断、分布、空值率、唯一性
│   ├── cross_ref()       # 跨表关联检查：外键完整性、孤儿记录
│   └── report()          # 生成探查报告（数据质量基线）
├── anomaly_detector.py   # 异常检测
│   ├── statistical()     # 统计异常（3σ、IQR）
│   └── pattern()         # 模式异常（日期格式不统一、编码混乱）
└── profiling_reports/    # 探查报告输出
```

#### 阶段 2：数据清洗（Data Cleaning）

| 问题类型 | 煤炭行业典型场景 | 技术方案 |
|---------|---------------|---------|
| 重复数据 | 同一供应商以多种名称出现 | 多级模糊匹配（见下方算法） |
| 缺失值 | 资质等级空、注册资本空、联系方式空 | 规则填充（关联表补全）+ 历史值填充 + 标记"待补全" |
| 格式不一致 | 日期多种格式、金额带/不带单位 | 正则表达式批量标准化 |
| 异常值 | 中标价 ¥50/吨（正常 ¥400-700）、数量为负 | 统计阈值检测 + 业务规则校验，标记疑似异常 |
| 编码错误 | GBK/UTF-8 混合、乱码 | 编码检测 + 统一转 UTF-8 |
| 过期数据 | 供应商已注销但仍在库、过期资质 | 关联工商数据核验 + 时间戳过期标记 |

**供应商去重算法（核心清洗环节）**：

```
输入: 供应商名称列表
    ↓
第 1 级: 精确匹配（去空格/括号后完全相同）→ 直接合并
    ↓
第 2 级: 规则匹配（去除"有限公司/股份/集团/责任"等后缀）→ 高置信度合并
    ↓
第 3 级: 模糊匹配（编辑距离 + Jaccard + 字符 n-gram）→ 置信度 >0.85 自动合并
    ↓
第 4 级: 属性交叉验证（同一地区 + 同一行业 + 近似名称）→ 置信度 >0.7 候选合并
    ↓
第 5 级: 人工确认（置信度 <0.7 的候选对）→ 人工标注
```

#### 阶段 3：数据标准化（Data Standardization）

将"各说各话"的数据统一成"同一种语言"。

| 标准化维度 | 具体内容 | 实现方式 |
|-----------|---------|---------|
| 实体标准化 | 供应商名称 → 统一社会信用代码 | 工商数据API对齐 + 内部映射表 |
| 煤种分类 | "动力煤"/"电煤"/"发电用煤" → GB/T 5751 标准编码 | 行业术语同义词表 |
| 地区编码 | "山西"/"山西省"/"晋" → GB/T 2260 行政区划代码 | 地区同义词表 + 层级映射 |
| 度量单位 | "吨"/"t"/"T" → 吨；"大卡"/"kcal" → kcal/kg | 单位转换规则引擎 |
| 价格口径 | 含税/不含税、坑口价/到厂价/离岸价 → 标注价格类型 | 标注价格类型 + 换算系数 |
| 时间格式 | 多种日期格式 → ISO 8601 | 正则表达式批量转换 |
| 金额单位 | "万元"/"万"/"1000000" → 统一为元 | 数值解析 + 单位换算 |
| 资质等级 | "特级"/"壹级"/"一级" → 统一编码 | 等级映射表 |

**行业术语同义词表**（`coal_terminology.yaml`）：

```yaml
coal_types:
  动力煤: [电煤, 发电用煤, 锅炉用煤, thermal_coal]
  炼焦煤: [焦煤, 冶金煤, coking_coal]
  无烟煤: [白煤, 硬煤, anthracite]
  褐煤: [brown_coal, lignite]

regions:
  山西省: [山西, 晋, SX]
  内蒙古自治区: [内蒙古, 蒙, NM]

price_types:
  坑口价: [mine-mouth price, 矿口价]
  到厂价: [delivered price, 含运费价]
  离岸价: [FOB, FOB价]

quality_params:
  热值: [发热量, 卡值, calorific_value, heat_value]
  灰分: [ash_content, 灰含量]
  硫分: [sulfur_content, 全硫, 含硫量]
  挥发分: [volatile_matter, 挥发份]
```

#### 阶段 4：数据加工富化（Data Enrichment）

在原始数据基础上计算衍生指标，增加数据价值。

| 衍生数据 | 计算逻辑 | 应用价值 |
|---------|---------|---------|
| 供应商信用评分 | 资质等级(30%) + 履约历史(25%) + 财务健康(20%) + 合规记录(15%) + 行业口碑(10%) | 供应商筛选和风险评估 |
| 价格指数 | 加权平均中标价（按煤种×地区×时间窗口），标准化为指数 | 市场趋势分析 |
| 市场集中度 | HHI 指数（前 N 大供应商市场份额平方和） | 反垄断分析、竞争格局 |
| 中标率 | 供应商历史投标次数 vs 中标次数 | 竞争力评估 |
| 供应商依赖度 | 某区域/某煤种对特定供应商的采购占比 | 供应链风险 |
| 价格偏离度 | 单次中标价 vs 同期同区域均价的偏离百分比 | 异常中标检测 |
| 供需平衡指数 | 招标需求量 vs 可供应量的比值 | 价格预测 |
| 季节性系数 | 历史各月价格 vs 年均价的比值 | 季节性趋势分析 |
| 替代供应商推荐 | 相同煤种+相近地区+相当资质的供应商列表 | 供应链韧性 |

**供应商信用评分模型**：

```
信用评分 = Σ(维度分 × 权重)

维度分计算:
├── 资质等级分: 特级=100, 一级=85, 二级=70, 三级=55, 无=30
├── 履约历史分: 中标项目按时交付率 × 100
├── 财务健康分: 注册资本分(40%) + 年营收分(30%) + 负债率分(30%)
├── 合规记录分: 无违规=100, 轻微违规=70, 严重违规=30, 黑名单=0
└── 行业口碑分: 合作方评价均分(1-5) × 20

最终: 0-100 分，≥90 优选供应商, 70-89 合格, 50-69 关注, <50 风险
```

#### 阶段 5：质量治理（Data Governance）

建立持续的质量保障体系，而非一次性清洗。

**质量评分模型**（每条数据记录）：

```
质量分 = 完整性(40%) + 准确性(30%) + 时效性(20%) + 一致性(10%)

完整性: 非空必填字段数 / 总必填字段数
准确性: 通过校验规则的字段数 / 总校验规则数
时效性: max(0, 1 - (当前日期 - 数据更新日期) / 最大有效期)
一致性: 与关联数据无矛盾的比例
```

**质量规则引擎**：

| 规则类别 | 示例规则 | 严重级别 |
|---------|---------|---------|
| 完整性 | 供应商名称、统一社会信用代码不能为空 | BLOCK（阻断发布） |
| 格式 | 统一社会信用代码必须是 18 位字母数字 | WARN（警告） |
| 范围 | 中标价必须在 100-2000 元/吨之间 | WARN |
| 逻辑 | 中标日期不能早于招标日期 | BLOCK |
| 关联 | 中标供应商必须在供应商库中存在 | BLOCK |
| 时效 | 供应商资质认证有效期不超过 3 年 | WARN |
| 唯一 | 同一招标项目只能有一条中标记录 | BLOCK |

**数据血缘追踪**：

```
原始数据记录
    ↓ [ETL Job #1: 时间戳]
清洗后记录 (标记: 清洗规则列表, 修改字段, 修改前后值)
    ↓ [ETL Job #2: 时间戳]
标准化记录 (标记: 标准化映射规则, 原始值→标准值)
    ↓ [ETL Job #3: 时间戳]
富化记录 (标记: 衍生字段计算公式, 输入字段)
    ↓ [Quality Gate]
质量评分 ≥ 70 → 发布到服务层
质量评分 < 70 → 进入"待治理"队列
```

**治理仪表盘指标**：

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| 整体质量分 | 全量数据的加权平均质量分 | < 80 |
| 阻断率 | 被 BLOCK 规则拦截的记录占比 | > 5% |
| 脏数据积压 | "待治理"队列中的记录数 | > 1000 |
| 新数据达标率 | 新入库数据质量分 ≥ 70 的比例 | < 90% |
| 修复响应时间 | 从发现质量问题到修复完成的平均时间 | > 24h |

#### 阶段 6：数据标注（Data Annotation）

为 AI 训练和高级分析准备标注数据。

| 标注类型 | 标注内容 | 用途 | 标注方式 |
|---------|---------|------|---------|
| 业务分类标注 | 招标项目类型（物资/工程/服务）、采购方式 | NL2SQL 训练、分类模型 | 规则自动标注 + 人工抽检 |
| 风险等级标注 | 供应商风险等级、项目风险标签 | 风险评估模型 | 模型预标注 + 专家审核 |
| 关系标注 | 供应商-竞争对手关系、母子公司关系、关联交易 | 知识图谱构建 | 算法发现 + 人工确认 |
| 时序事件标注 | 价格突变事件（政策/产能/突发事件） | 时序预测模型 | 专家标注（事件类型+影响范围+持续时间） |
| 查询意图标注 | 自然语言查询 → SQL 的配对数据 | NL2SQL 模型微调 | 专家编写 + 模型辅助生成 |
| 质量评分标注 | 对数据记录的人工质量评级（优/良/差） | 质量模型训练 | 抽样人工标注 |

**NL2SQL 训练数据标注流程**：

```
第 1 步: 收集真实查询
  - 从日志中提取用户历史查询（脱敏后）
  - 专家编写覆盖各场景的标准查询集

第 2 步: 生成 SQL 标注
  - 数据库专家编写对应的 SQL
  - 标注 SQL 涉及的表、字段、意图类型

第 3 步: 多样性扩展
  - 同义改写: "山西动力煤价格" = "山西省电煤行情" = "晋地区发电用煤报价"
  - 复杂度分层: 简单查询 / 聚合查询 / 多表关联 / 嵌套子查询

第 4 步: 质量校验
  - 自动执行 SQL 验证语法正确性
  - 对比多条 SQL 的结果一致性
  - 人工抽检 ≥ 20%

目标: 覆盖 8 大数据集 × 5 种查询复杂度 × 10+ 同义表达 = 400+ 高质量标注对
```

#### 文档数据智能抽取

非结构化文档（PDF/Word）的抽取独立于上述结构化流水线，流程为：

1. **文档解析**: PDF/Word → 结构化文本 + 表格提取
2. **字段抽取**: AI 识别并提取关键信息（项目名称、金额、技术参数、资质要求、煤质指标等）
3. **表格识别**: OCR + 表格结构还原 → 转为结构化记录
4. **关联入库**: 抽取结果入库，与已有结构化数据关联

#### 实时流处理

对接实时数据源（价格行情、物流轨迹、库存变动），通过流式管道实时入库，走简化的清洗+标准化流程（跳过人工环节）。

### 2.4 合规脱敏层

1. **敏感数据识别**: 自动识别商业秘密（报价、利润率）、个人隐私（联系人信息）、价格敏感信息
2. **分级脱敏**: 根据订阅等级决定是否展示原始价格、供应商联系方式等敏感字段
3. **合规审计日志**: 记录每次数据访问的调用方、查询内容、返回数据范围
4. **数据水印**: 返回数据中嵌入调用方标识水印，防止数据转售和泄露追溯
5. **行业合规**: 满足煤炭行业监管要求（数据出境限制、国企数据管理规范等）

## 3. 变现模式 A：MCP 数据服务

### 3.1 查询处理架构：双通道模式

MCP 服务采用**结构化数据 NL2SQL + 非结构化数据 RAG** 的双通道查询架构：

```
用户自然语言问题
        ↓
    MCP 路由层
   ┌────┴────┐
   ↓         ↓
结构化通道   非结构化通道
(NL2SQL)     (RAG)
   ↓         ↓
数据库查询   向量检索 + LLM 生成
   ↓         ↓
 表格数据    文档/报告内容
```

#### 通道 1：结构化数据 → NL2SQL

针对供应商库、招投标记录、价格数据、煤质数据等结构化数据。

**核心流程**：自然语言 → Schema 感知的 NL2SQL → SQL 安全校验 → 沙箱执行 → 结果后处理

| 环节 | 实现方式 | 作用 |
|------|---------|------|
| Schema 感知 | 维护数据字典（表名、字段、枚举值、同义词映射） | 让 LLM 生成正确的 SQL |
| 同义词映射 | "动力煤"→ coal_type='thermal', "热值"→ heat_value | 解决行业术语差异 |
| SQL 安全校验 | 白名单表/字段 + 只读模式 + 查询超时 + 结果行数限制 | 防止危险操作 |
| 结果后处理 | 自动聚合摘要 + 异常值标注 + 置信度评分 | 提升可读性 |

**示例**：
```
输入: "去年山西地区中标价最低的5个动力煤项目是哪些？"
      ↓
数据字典注入: tables=[bidding], columns=[region, coal_type, winning_price, project_name, bid_date]
同义词映射: "去年"→ year=2025, "最低"→ ORDER BY ASC LIMIT 5
      ↓
生成 SQL: SELECT project_name, winning_price, bid_date
         FROM bidding
         WHERE region='山西' AND coal_type='动力煤' AND year=2025
         ORDER BY winning_price ASC LIMIT 5
      ↓
安全校验: ✓ 只读查询, ✓ 白名单表, ✓ 结果行数≤5
      ↓
执行返回: [{project_name: "XX煤矿采购", winning_price: 520, ...}, ...]
```

#### 通道 2：非结构化数据 → RAG

针对招标文件、合同文本、地质报告、政策法规等文档数据。

**核心流程**：自然语言 → 向量检索 → 相关文档片段 → LLM 生成答案 → 引用溯源

| 环节 | 实现方式 | 作用 |
|------|---------|------|
| 文档切片 | 按段落/章节切分 + 表格保留结构 | 保证检索精度 |
| 向量索引 | Embedding + 向量数据库（复用 EAI Flow 知识工厂） | 语义检索 |
| 混合检索 | 向量检索 + BM25 关键词检索 + 元数据过滤 | 提高召回率 |
| 引用溯源 | 返回结果附带源文档出处 | 增强可信度 |

**示例**：
```
输入: "山西地区煤矿安全验收的标准要求是什么？"
      ↓
向量检索: 在法规政策库和地质报告库中检索 top-5 相关片段
      ↓
LLM 生成: "根据《山西省煤矿安全生产条例》第X条... [引用: doc_123_p5]"
```

#### 通道 3：智能路由（`coal_hybrid_query`）

自动判断问题应该走 NL2SQL 还是 RAG，或两者组合：
- 涉及数字、趋势、对比、排名 → 路由到 NL2SQL
- 涉及标准、规范、原因、解释 → 路由到 RAG
- 涉及"为什么降价了"类复合问题 → NL2SQL 获取数据 + RAG 获取行业洞察 → 组合回答

### 3.2 多轮对话能力

#### 两层级上下文管理

**层级 1：AI 端上下文（已有）**

调用方 AI（Claude/GPT 等）自带对话上下文，自行处理指代消解（"它"、"那"、"去年同期"）。大部分场景靠这个就够了。

**层级 2：MCP 服务端会话上下文（新增）**

对于需要服务端感知上下文的场景（查询精炼、数据钻取），MCP Tool 支持可选的 `session_id` 参数：

```python
coal_nl2sql(
    question="和去年同期比呢？",
    session_id="sess_abc123"  # 可选，用于多轮上下文
)
```

服务端会话上下文维护：

| 上下文类型 | 存储内容 | 用途 |
|-----------|---------|------|
| 查询历史 | 最近 N 轮的自然语言 + 生成的 SQL + 结果摘要 | 指代消解、查询修正 |
| 实体记忆 | 本轮对话中提到的地区、煤种、时间范围 | 默认参数填充 |
| 分析维度 | 当前分析的角度（价格/供应商/质量） | 保持分析连贯性 |
| 数据快照 | 最近一次查询结果的统计摘要（非原始数据） | 对比分析时快速引用 |

**多轮对话示例**：
```
第 1 轮: "山西动力煤最近中标价怎么样？"
  session 存入: {region: "山西", coal_type: "动力煤", metric: "中标价", time: "最近"}
  → 返回: "近3月平均中标价 ¥580/吨，环比下降3%"

第 2 轮: "和去年同期比呢？"
  从 session 取出上下文 → 补全参数
  → 返回: "2025年 ¥580 vs 2024年 ¥610，同比下降4.9%"

第 3 轮: "那内蒙古的呢？"
  从 session 取出 → 替换 region='内蒙古'，保留其他参数
  → 返回: "内蒙古 ¥520 vs 山西 ¥580，低10.3%"
```

#### 多轮对话安全防护

| 风险 | 场景 | 防护 |
|------|------|------|
| 上下文漂移 | 多轮后查询偏离原始意图 | 会话最多保留 20 轮，超时 30 分钟自动清理 |
| SQL 注入叠加 | 恶意利用多轮修正逐步构造危险查询 | 每轮 SQL 独立做安全校验，不信任历史上下文 |
| 结果过时 | 会话期间数据已更新 | 每轮重新查询，不缓存原始结果 |
| 隐私泄露 | 会话中包含敏感信息 | 会话结束时清除，不持久化 |
