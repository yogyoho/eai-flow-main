# 煤炭行业数据资产建设与变现策略设计

> **日期**: 2026-05-26
> **状态**: 待实施
> **范围**: 煤炭招投标/供应链平台数据资产化与变现

---

## 1. 背景与目标

煤炭领域招投标和供应链互联网平台积累了海量数据，包括供应商、招标、采购、财务、供应链等数据。本设计定义如何将这些数据构建为高质量数据集，并通过 MCP 服务订阅、数据集售卖、AI 训练平台三种模式实现数据资产变现。

### 现状

- **数据存储**: 混合存储（结构化数据库 + 非结构化文档）
- **数据量级**: 100万-1000万条记录 + 大量 PDF/Word 文档
- **变现方向**: AI 数据服务（MCP/API）为核心，数据集和训练平台为辅
- **策略**: 渐进式——先嵌入 EAI Flow 平台验证，后独立扩展

## 2. 数据集建设架构

### 2.1 数据分层架构

```
原始数据层 → 清洗标注层 → 合规脱敏层 → 结构化数据层 → 服务发布层
    ↓             ↓             ↓              ↓              ↓
  数据采集      质量控制       安全合规       数据资产目录     MCP/API/数据集
                                                        ↕
                                                    实时数据流
```

### 2.3 清洗标注层

1. **文档智能抽取**: AI 从 PDF/Word 提取结构化字段（项目名称、金额、技术参数、资质要求、煤质指标等）
2. **实体标准化**: 统一供应商名称（去除别名/曾用名）、煤种分类（国标编码）、地区编码、度量单位
3. **关联图谱构建**: 供应商-项目-中标-交付-付款 完整关系链，支持图谱查询
4. **质量评分**: 每条数据计算完整性、准确性、时效性三维评分，标注数据可信度
5. **实时流处理**: 对接实时数据源（价格行情、物流轨迹、库存变动），通过流式管道实时入库

### 2.4 合规脱敏层

1. **敏感数据识别**: 自动识别商业秘密（报价、利润率）、个人隐私（联系人信息）、价格敏感信息
2. **分级脱敏**: 根据订阅等级决定是否展示原始价格、供应商联系方式等敏感字段
3. **合规审计日志**: 记录每次数据访问的调用方、查询内容、返回数据范围
4. **数据水印**: 返回数据中嵌入调用方标识水印，防止数据转售和泄露追溯
5. **行业合规**: 满足煤炭行业监管要求（数据出境限制、国企数据管理规范等）

## 3. 变现模式 A：MCP 数据服务

### 3.1 查询处理架构：双通道模式

MCP 服务采用**结构化数据 NL2SQL + 非结构化数据 RAG** 的双通道查询架构：

```
用户自然语言问题
        ↓
    MCP 路由层
   ┌────┴────┐
   ↓         ↓
结构化通道   非结构化通道
(NL2SQL)     (RAG)
   ↓         ↓
数据库查询   向量检索 + LLM 生成
   ↓         ↓
 表格数据    文档/报告内容
```

#### 通道 1：结构化数据 → NL2SQL

针对供应商库、招投标记录、价格数据、煤质数据等结构化数据。

**核心流程**：自然语言 → Schema 感知的 NL2SQL → SQL 安全校验 → 沙箱执行 → 结果后处理

| 环节 | 实现方式 | 作用 |
|------|---------|------|
| Schema 感知 | 维护数据字典（表名、字段、枚举值、同义词映射） | 让 LLM 生成正确的 SQL |
| 同义词映射 | "动力煤"→ coal_type='thermal', "热值"→ heat_value | 解决行业术语差异 |
| SQL 安全校验 | 白名单表/字段 + 只读模式 + 查询超时 + 结果行数限制 | 防止危险操作 |
| 结果后处理 | 自动聚合摘要 + 异常值标注 + 置信度评分 | 提升可读性 |

**示例**：
```
输入: "去年山西地区中标价最低的5个动力煤项目是哪些？"
      ↓
数据字典注入: tables=[bidding], columns=[region, coal_type, winning_price, project_name, bid_date]
同义词映射: "去年"→ year=2025, "最低"→ ORDER BY ASC LIMIT 5
      ↓
生成 SQL: SELECT project_name, winning_price, bid_date
         FROM bidding
         WHERE region='山西' AND coal_type='动力煤' AND year=2025
         ORDER BY winning_price ASC LIMIT 5
      ↓
安全校验: ✓ 只读查询, ✓ 白名单表, ✓ 结果行数≤5
      ↓
执行返回: [{project_name: "XX煤矿采购", winning_price: 520, ...}, ...]
```

#### 通道 2：非结构化数据 → RAG

针对招标文件、合同文本、地质报告、政策法规等文档数据。

**核心流程**：自然语言 → 向量检索 → 相关文档片段 → LLM 生成答案 → 引用溯源

| 环节 | 实现方式 | 作用 |
|------|---------|------|
| 文档切片 | 按段落/章节切分 + 表格保留结构 | 保证检索精度 |
| 向量索引 | Embedding + 向量数据库（复用 EAI Flow 知识工厂） | 语义检索 |
| 混合检索 | 向量检索 + BM25 关键词检索 + 元数据过滤 | 提高召回率 |
| 引用溯源 | 返回结果附带源文档出处 | 增强可信度 |

**示例**：
```
输入: "山西地区煤矿安全验收的标准要求是什么？"
      ↓
向量检索: 在法规政策库和地质报告库中检索 top-5 相关片段
      ↓
LLM 生成: "根据《山西省煤矿安全生产条例》第X条... [引用: doc_123_p5]"
```

#### 通道 3：智能路由（`coal_hybrid_query`）

自动判断问题应该走 NL2SQL 还是 RAG，或两者组合：
- 涉及数字、趋势、对比、排名 → 路由到 NL2SQL
- 涉及标准、规范、原因、解释 → 路由到 RAG
- 涉及"为什么降价了"类复合问题 → NL2SQL 获取数据 + RAG 获取行业洞察 → 组合回答

### 3.2 多轮对话能力

#### 两层级上下文管理

**层级 1：AI 端上下文（已有）**

调用方 AI（Claude/GPT 等）自带对话上下文，自行处理指代消解（"它"、"那"、"去年同期"）。大部分场景靠这个就够了。

**层级 2：MCP 服务端会话上下文（新增）**

对于需要服务端感知上下文的场景（查询精炼、数据钻取），MCP Tool 支持可选的 `session_id` 参数：

```python
coal_nl2sql(
    question="和去年同期比呢？",
    session_id="sess_abc123"  # 可选，用于多轮上下文
)
```

服务端会话上下文维护：

| 上下文类型 | 存储内容 | 用途 |
|-----------|---------|------|
| 查询历史 | 最近 N 轮的自然语言 + 生成的 SQL + 结果摘要 | 指代消解、查询修正 |
| 实体记忆 | 本轮对话中提到的地区、煤种、时间范围 | 默认参数填充 |
| 分析维度 | 当前分析的角度（价格/供应商/质量） | 保持分析连贯性 |
| 数据快照 | 最近一次查询结果的统计摘要（非原始数据） | 对比分析时快速引用 |

**多轮对话示例**：
```
第 1 轮: "山西动力煤最近中标价怎么样？"
  session 存入: {region: "山西", coal_type: "动力煤", metric: "中标价", time: "最近"}
  → 返回: "近3月平均中标价 ¥580/吨，环比下降3%"

第 2 轮: "和去年同期比呢？"
  从 session 取出上下文 → 补全参数
  → 返回: "2025年 ¥580 vs 2024年 ¥610，同比下降4.9%"

第 3 轮: "那内蒙古的呢？"
  从 session 取出 → 替换 region='内蒙古'，保留其他参数
  → 返回: "内蒙古 ¥520 vs 山西 ¥580，低10.3%"
```

#### 多轮对话安全防护

| 风险 | 场景 | 防护 |
|------|------|------|
| 上下文漂移 | 多轮后查询偏离原始意图 | 会话最多保留 20 轮，超时 30 分钟自动清理 |
| SQL 注入叠加 | 恶意利用多轮修正逐步构造危险查询 | 每轮 SQL 独立做安全校验，不信任历史上下文 |
| 结果过时 | 会话期间数据已更新 | 每轮重新查询，不缓存原始结果 |
| 隐私泄露 | 会话中包含敏感信息 | 会话结束时清除，不持久化 |

```
│   ├── desensitizer.py           # 脱敏处理
│   ├── watermark.py              # 数据水印
│   └── audit_logger.py           # 审计日志
├── mcp_server/
│   └── coal_data_server.py       # 煤炭数据 MCP Server（注册所有 Tool）
├── billing/                      # 计费系统
│   ├── subscription.py           # 订阅管理
│   ├── quota.py                  # 调用配额
│   └── metering.py               # 用量计量
├── routers.py                    # API 路由
├── service.py                    # 业务逻辑
└── schemas.py                    # 数据模型
```

MCP Server 注册到 `extensions_config.json`，AI Agent 自动获得数据查询能力。外部应用通过标准 MCP 协议接入。
