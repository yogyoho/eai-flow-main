# 两层 DAG 工作流引擎重构方案

## 问题诊断

当前工作流编辑器将 6 种节点类型（阶段、审核、条件、AI 生成、汇聚、子流程）全部平铺在一个画布上。但在工程报告项目中，**阶段是容器，不是节点**。真实的层次结构是：

```
可行性研究阶段
├── 任务：资料收集（AI 生成）
├── 任务：初稿编写（AI 生成）
├── 任务：内部审核（审核，退回则循环）
└── 任务：阶段成果提交
    │
    ▼ 阶段完成
初步设计阶段
├── 任务：方案设计（AI 生成 + 人工修改）
├── 任务：技术审核
...
```

## 核心设计：两层 DAG

### 上层：阶段 DAG（Phase Graph）

粗粒度的里程碑编排。节点类型只有两种：

| 节点类型 | 说明 |
|---------|------|
| **PhaseNode**（阶段） | 容器，内部包含一个任务 DAG。有名称、团队、预计工期、章节范围 |
| **ConditionNode**（条件） | 阶段之间的分支判断。表达式引用阶段输出或项目属性 |

阶段之间连线表示依赖和流转方向。条件节点可以有 `true`/`false` 两条出边。

### 下层：任务 DAG（Task Graph）

每个阶段内部包含一个细粒度任务图。任务节点类型：

| 节点类型 | 说明 |
|---------|------|
| **AIGenerateTask**（AI 生成） | 调用 LLM 生成章节内容。绑定章节、提示词模板 |
| **ManualEditTask**（人工编写/修改） | 人工操作节点。分配给用户/角色。完成后手动推进 |
| **ReviewTask**（审核） | 审核节点。支持多种签核模式。绑定审核人 |
| **MergeTask**（汇聚） | 等待多个上游任务完成后合并 |
| **NotifyTask**（通知） | 发送通知（邮件、站内信、IM） |

### 驳回路由规则

审核不通过时的回退行为：

1. **默认行为**：退回到当前阶段的第一个任务，重新走整个阶段流程
2. **指定回退**：在审核节点的出边上标注 `rejected`，指向同阶段内的任意任务节点
3. **跨阶段回退**：退回上一个阶段（阶段之间默认的驳回行为）
4. **手动回退**：项目负责人可以手动将项目整体回退到任意阶段

数据结构扩展：

```json
{
  "phaseGraph": {
    "nodes": [],
    "edges": []
  },
  "taskGraphs": {
    "phase-id-1": {
      "nodes": [],
      "edges": []
    }
  }
}
```

审核边的标签约定：
- `approved` — 通过（默认）
- `rejected` — 不通过，沿此边回退
- 如果审核节点没有 `rejected` 出边，默认退到阶段起点

## 审核签核模式

每个审核节点（ReviewTask）支持四种模式：

| 模式 | 含义 | 通过条件 |
|------|------|---------|
| **会签 (all)** | 所有审核人必须通过 | 全部审批 |
| **或签 (any)** | 任一审核人通过即可 | 至少 1 人审批 |
| **比例签 (ratio)** | 达到指定比例 | 通过数 / 总数 ≥ 阈值 |
| **顺序签 (sequential)** | 按顺序逐级审批 | 指定顺序全部通过 |

审核节点数据结构：

```typescript
interface ReviewTaskData {
  mode: "all" | "any" | "ratio" | "sequential";
  ratioThreshold?: number;           // 比例签的阈值 (0.0-1.0)
  reviewers: ReviewerBinding[];      // 审核人绑定列表
  allowDelegation?: boolean;         // 是否允许转审
  deadlineHours?: number;            // 审核时限（小时）
  rollbackTarget?: string;           // 驳回目标节点 ID（不填则默认退回阶段起点）
}

interface ReviewerBinding {
  type: "role" | "position" | "user";
  value: string;                     // 角色 key / 岗位 code / 用户 ID
  label: string;                     // 显示名
  required: boolean;                 // 是否必须有实体人（无人则阻断）
  order?: number;                    // 顺序签的审批顺序
}
```

后端审核逻辑调整（`_execute_review` 方法扩展）：

```
输入：ReviewTaskData
1. 解析 ReviewerBinding[] → 找到具体审核人
2. 创建 PhaseReview 记录
3. 根据 mode 等待审核完成：
   - all: 等所有审核人完成，全部 approved 才通过
   - any: 等任一审核人 approved 就通过
   - ratio: 等通过比例 ≥ threshold
   - sequential: 按 order 序列化，逐个等待
4. 任一 reject → 沿 rejected 边回退
```

## 节点绑定 (Node Binding)

任务节点支持绑定到**角色**、**岗位**、**用户**三种维度，优先级从低到高：

```
角色 → 岗位 → 用户
（粗）            （细）
```

绑定数据结构：

```typescript
interface NodeAssignment {
  // 角色绑定：如 "writer"（撰写人）、"reviewer"（审核人）
  roles?: { roleKey: string; count: number; label: string }[];
  
  // 岗位绑定：如 "部门负责人"、"项目经理"
  positions?: { positionCode: string; label: string }[];
  
  // 用户绑定：指定具体用户
  users?: { userId: string; username: string }[];
  
  // 绑定策略
  strategy: "require_all" | "require_any" | "prefer_role";
  
  // 无人时的处理
  onNoAssignee: "block" | "skip" | "notify_admin";
}
```

**解析优先级**：
1. 如果有 `users` 绑定 → 直接使用指定用户
2. 如果有 `positions` 绑定 → 查找项目中担任该岗位的成员
3. 如果有 `roles` 绑定 → 查找项目中分配了该角色的成员
4. 全部为空 + `onNoAssignee: "block"` → 阻断，通知项目管理员

**绑定生效时机**：
- 模板编辑时：预设绑定规则（"需要一个审批人角色的成员"）
- 项目创建时：继承模板的绑定规则
- 运行时：根据项目实际成员动态解析具体的执行人

## 编辑器 UX 改动

### 阶段级画布（主视图）

```
┌─────────────────────────────────────────────────────────┐
│ 工具栏：[+阶段] [+条件] [保存] [验证]                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌──────────────┐     ┌──────────────┐                  │
│   │ 可行性研究    │────▶│ 初步设计      │                  │
│   │ 任务数: 4    │     │ 任务数: 6    │                  │
│   │ 团队: 地质组  │     │ 团队: 设计组  │                  │
│   └──────────────┘     └──────────────┘                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

- 双击阶段节点 → 进入该阶段的任务级画布
- 阶段节点显示：名称、任务数、团队名、状态（未开始/进行中/已完成）
- 条件节点放在阶段之间，控制流转

### 任务级画布（双击阶段进入）

```
┌─────────────────────────────────────────────────────────┐
│ ← 返回阶段视图 ｜ 阶段：可行性研究      [+添加任务]        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────┐   ┌─────────┐   ┌─────────────┐            │
│  │ AI 生成  │──▶│ 人工修改 │──▶│ 技术审核      │            │
│  │ (第1章)  │   │ (第1章)  │   │ 角色: reviewer│            │
│  └─────────┘   └─────────┘   │ 模式: all      │            │
│                               └──┬──────┬────┘            │
│                          approved │      │ rejected       │
│                                   ▼      ▼               │
│                              ┌──────┐ ┌─────────┐       │
│                              │ 提交  │ │ AI 生成  │       │
│                              └──────┘ │ (重新写)  │       │
│                                       └─────────┘       │
└─────────────────────────────────────────────────────────┘
```

- 任务节点小一号，连线规则同 DAG
- 审核节点有 `approved`（绿边）和 `rejected`（红边）两种出边
- 右侧属性面板：选中的阶段/任务显示配置

## 数据迁移

当前 `graph_json` 结构（平铺式）需要迁移到新结构（两层式）。迁移策略：

1. **数据兼容**：`DynamicGraphWorkflow` 同时支持新旧两种 `graph_json` 格式，通过 `version` 字段区分
2. **自动迁移**：前端加载旧模板时，尝试自动转换（把 phase 节点作为阶段容器，其他节点归入单独阶段的内部任务）
3. **手动重编**：复杂拓扑由管理员手动重建

```python
# 后端兼容判断
if graph.get("version") == 2:
    return await self._execute_v2(graph)
else:
    return await self._execute_v1(graph)  # 当前逻辑
```
