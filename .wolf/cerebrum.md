## Key Learnings

- **Project:** eai-flow-main
- **Description:** English | [中文](./README_zh.md) | [日本語](./README_ja.md) | [Français](./README_fr.md) | [Русский](./README_ru.md)
- **Database session providers:** Always reuse a shared engine pool. Never create+dispose engines per request — the overhead is catastrophic. Use `pool_pre_ping=True` instead of `SELECT 1` validation on every request.
- **Nginx upstream keepalive:** Always add `keepalive N` to upstream blocks and set `proxy_set_header Connection ''` on location blocks to reuse TCP connections to backend.
- **MCP skill tool filtering:** Skills with `allowed-tools` declarations in SKILL.md cause `filter_tools_by_skill_allowed_tools()` to strip ALL tools not in the allowed list, including MCP tools. If no skills are enabled, the filter returns None (allow-all). Disabled/renamed skills don't count.
- **MCP lazy init + caching:** `get_cached_mcp_tools()` caches MCP tools after first successful initialization. The cache invalidates on `extensions_config.json` mtime changes. First agent run may have empty MCP tools if init hasn't completed yet.
- **MCP Word server path issue:** Office-Word-MCP-Server's `create_document` saves files relative to the MCP server's CWD (which is `/app/backend/` in Docker), NOT the container's virtual path system (`/mnt/user-data/outputs/`). The agent cannot directly place files into the virtual path system via MCP tools.
- **Docker container names:** This project uses `docker compose -p eai-docker`, so containers are named `eai-docker-*` (but actual names are `deer-flow-gateway`, `deer-flow-frontend`, `deer-flow-nginx`). Always `docker ps` to verify.
- **Admin email:** The admin account in the Docker DB is `admin@eai-flow.com` (not `admin@eai.local`).
- **BlockNote provider type cast:** HocuspocusProvider has `awareness: Awareness | null` but BlockNote's `CollaborationOptions.provider` expects `{ awareness?: Awareness }`. Since `y-protocols/awareness` is not hoisted to top-level node_modules, use `provider as any` cast to bridge the types.
- **BlockNote CSS imports:** BlockNote requires two CSS files: `@blocknote/shadcn/dist/style.css` and `@blocknote/react/dist/style.css`. The CSS files are only in `dist/`, not at the package root.
- **Turbopack + transpilePackages breaks subpath exports:** Adding a package with pre-compiled `dist/` to `transpilePackages` in Next.js/Turbopack causes subpath imports like `@scope/pkg/locales` to resolve incorrectly. Turbopack tries source resolution and ignores the `exports` map, resulting in undefined imports at runtime. Only use `transpilePackages` for packages that actually need compilation (source with JSX/TS, no dist).
- **BlockNote AI dictionary:** The `@blocknote/xl-ai` AIExtension requires `editor.dictionary.ai` to be set. The dictionary must be merged with the base dictionary (`@blocknote/core/locales`): `{ ...coreEn, ai: aiEn }`. Using only `{ ai: aiEn }` replaces the entire base dictionary and loses core UI translations.
- **BlockNote lifecycle:** `useCreateBlockNote` manages editor lifecycle automatically. Do NOT call `editor.destroy()` manually in a useEffect cleanup.
- **BlockNote hooks API:** `useCreateBlockNote` accepts options as first arg and a deps array as second arg (like useMemo). The editor is recreated when deps change.
- **Hocuspocus 2.x hooks:** No `onDestroyDocument` — use `afterUnloadDocument({ documentName })` instead. The Document type from `@hocuspocus/server` has `connections` (Map) for checking active users.
- **Gateway vs Extensions user ID bridge:** The Hocuspocus collab server receives Gateway user IDs (from JWT sub) which differ from the Extensions PostgreSQL user IDs. The `bridgeGatewayUser()` function in persistence.ts calls Gateway API to get email, then looks up the extensions user by email.
- **BlockNote AI stream format:** The AI SDK's `DefaultChatTransport` expects SSE chunks with specific types. `{ type: "tool-call" }` is INVALID — use `{ type: "tool-input-start" }` followed by `{ type: "tool-input-available", toolCallId, toolName, input }`. The `@blocknote/xl-ai` AIExtension handles `tool-input-available` to apply tracked changes.
- **BlockNote AI menu review items:** The `@blocknote/xl-ai` AIMenu component does NOT have built-in accept/reject buttons. The `items` callback MUST return accept/reject items for "user-reviewing" status and retry/cancel items for "error" status. Call `ai.acceptChanges()` and `ai.rejectChanges()` from `editor.getExtension(AIExtension)`.
- **Yjs snapshot is binary, not text:** `CollabDocument.yjs_doc` and `CollabVersion.snapshot` store Yjs CRDT binary state, NOT UTF-8 text. Decoding with `.decode("utf-8")` produces garbled output. For diff/summary, use `snapshot_text` (markdown) column instead, populated by frontend's `editor.blocksToMarkdownLossy()`.
- **Dashboard API auth:** Dashboard endpoints use `CurrentUserWithAccess` / `DashboardUser` (requires `system:access` permission). All dashboard routes are under `/api/extensions/dashboard/`.
- **Workflow graph_json is flexible:** DAG node data is stored in `WorkflowDefinition.graph_json` (JSONB). New fields like `required_roles` don't need model migrations — they live inside the JSON. Only top-level fields like `org_bindings` need ALTER TABLE.
- **PhaseConfigPanel is called from WorkflowEditor:** The PhaseConfigPanel is rendered in WorkflowEditor.tsx with `{ data, onUpdate }` props. New props must be optional to avoid breaking the call site.
- **react-day-picker CustomComponents type:** The `Calendar` wrapper from Shadcn/ui uses `react-day-picker` v9 which has `components.DayContent` type issues. Prefer using `modifiers` + `modifiersStyles` for visual customization instead of custom components.
- **Tailwind v4 border FOUC:** In Tailwind CSS v4, the `.border` utility class only sets `border-width` and `border-style` in `@layer utilities`, but `border-color` relies on `@layer base { * { border-color: var(--border) } }`. During Next.js client-side navigation with Turbopack HMR, the base layer can briefly not apply, causing fallback to `currentColor` (black). Fix: always pair `border` with `border-border` to put border-color in the utilities layer (higher cascade priority). Also add `@source inline("border-border")` to globals.css to ensure the utility is generated.
- **DB migration pattern:** New tables/columns use `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` in `database.py`'s `migrate_db()`. No Alembic — just raw SQL in the async init function.
- **React derived state pattern:** When a parent manages a list (e.g., React Flow nodes) and a child needs to display/edit a selected item, NEVER store a snapshot of the selected item as independent state. Store only the selected ID and derive the item from the latest array via `useMemo`. Otherwise, edits to the item via state setters won't reflect in the child — it keeps reading the stale snapshot. This caused WorkflowEditor's property panel inputs to be non-functional.
- **Template publish/unpublish semantics:** `is_template` means "this IS a template" (structural flag). `template_status` means the publication state (draft/published/pending_approval/rejected). When unpublishing, only change `template_status` back to "draft" — do NOT set `is_template=false`. The template list filters by `is_template=true`, so clearing it makes the template vanish from the list even though graph data is intact in the DB.
- **Permission gate `project:create`:** Removed from default "user" role in `_ROLE_DEFAULTS`. The `_ensure_role` function auto-resets DB roles that have drifted (extra permissions beyond coded defaults). Regular users need explicit permission or admin role to create projects.
- **`require_super_admin` dependency:** Added to auth middleware for operations requiring system-level admin (workflow definition edit/delete/publish). Checks `role.is_system` or wildcard `"*"` in permissions.
- **Phase-scoped chapter access:** `ProjectChapter.phase_node` column (VARCHAR(100), nullable) tracks which workflow phase a chapter belongs to. Set during `init_phase` activity using `chapter_range` from workflow graph. Non-owner/manager users can only edit chapters in the current active phase.
- **Activity log pattern:** `ActivityLog` model tracks project actions (who/what/when). Use `log_activity()` helper from routers — pass db session, project_id, user_id, action string. Avoids N+1 queries by keeping the helper lightweight (just `db.add`).
- **Batch query pattern for list endpoints:** Avoid per-item queries in list endpoints. Use `group_by` queries to batch-fetch counts (chapters, members, completed chapters) in one SQL statement per metric, then join in Python.
- **Custom roles need system:access:** All extension API endpoints (projects, workflow, dashboard, etc.) require `system:access` permission. When creating custom roles, always include `system:access` + `kb:read` + `doc:read` as base permissions. The default "user" role has these, but custom roles do not inherit them automatically.
- **Project chapters have no REST creation endpoint:** `ProjectChapter` rows are created via `_import_template_outline()` (from ExtractionTemplate) or `copy-from` endpoint, not via individual POST. To create chapters for testing, insert directly into DB with `created_at`, `updated_at`, `word_count_current` fields.
- **Departments API returns nested tree:** `GET /api/extensions/departments` returns `{ departments: [rootNode] }` where children are nested inside `rootNode.children[]`. Must flatten the tree to find specific departments by name.
- **Approval endpoint is `/submit-approval`** (not `/approval/submit`): The project approval submission uses `POST /projects/{id}/submit-approval` with `ApprovalSubmitRequest` schema requiring `steps[].step_order` and `steps[].reviewer_id` (UUID).
- **Output API uses FormData, not JSON:** `POST /api/extensions/output/generate` expects `source`, `format`, `layout_template_id` as Form fields, plus optional `file` upload OR `content` string for markdown source. Not a JSON body.
- **DB migration pattern for new columns:** When adding a column to an existing table's SQLAlchemy model, you MUST also add it to the raw SQL CREATE TABLE in `database.py` AND add an `ALTER TABLE ADD COLUMN IF NOT EXISTS` for existing databases. Just updating the model is not enough — the raw SQL runs first on fresh DBs.
- **Comment system is fully functional:** `CollabComment` model + `CommentService` CRUD + threaded replies (parent_id) + resolve/reopen. API: `/docmgr/documents/{id}/comments` (GET/POST), `/docmgr/comments/{id}` (PUT/DELETE/resolve/reopen).
- **Version system is fully functional:** `CollabVersion` + `VersionService` with create/list/diff(difflib.SequenceMatcher)/restore. API: `/docmgr/documents/{id}/versions` (GET/POST), diff (GET ?from=&to=), restore (POST).
- **Reminder service triggers via endpoint:** `POST /api/extensions/dashboard/check-reminders` scans `ProjectTimeline` entries and milestones, creates `Notification` rows with urgency classification (overdue/urgent/warning/notice) and dedup.
- **CSRF token for browser console API testing:** Get from cookie `csrf_token`, pass as `X-CSRF-Token` header. Cookie name is literally `csrf_token` (not `__Host-csrf`).
- **Template list endpoint:** `GET /api/extensions/output/templates` returns `{items: [...], total: N}` (not `/layout-templates`).
- **Temporal sandbox incompatibility:** Temporal's default sandboxed workflow runner cannot import FastAPI (sniffio._ThreadLocal restriction). Must use `UnsandboxedWorkflowRunner()` from `temporalio.worker` — import is `from temporalio.worker import UnsandboxedWorkflowRunner`, NOT from `temporalio.worker.workflow_sandbox`.
- **Temporal in Docker needs TEMPORAL_URL:** Gateway runs in its own container, so `localhost:7233` points to the gateway itself. Must set `TEMPORAL_URL=eai-flow-temporal:7233` in docker-compose-dev.yaml. The temporal server compose file is `docker/docker-compose.temporal.yaml` and must be combined with `docker-compose.extensions.yaml` (which defines postgres-ext).
- **Temporal requires PostgreSQL user:** Temporal auto-setup needs a `temporal` PostgreSQL user with SUPERUSER and a `temporal` database. Must create before starting the server.
- **docker compose restart vs up:** `docker compose restart` does NOT re-read compose file changes (new env vars, etc). Must use `docker compose up -d <service>` to recreate the container with updated config.
- **DB session scoping in FastAPI endpoints:** When using `Depends(get_db)` in FastAPI, the DB session is tied to the request lifecycle. If you open a separate `get_db_context()` inside the endpoint, the `Depends(get_db)` session may be closed by middleware. For endpoints that need full control, remove `db: AsyncSession = Depends(get_db)` and use `async with get_db_context() as db:` inside the function body instead.
- **Workflow-status API includes graph_json:** The `GET /workflow/projects/{id}/workflow-status` endpoint returns `graph_json` (the full DAG structure) along with node statuses. Frontend should use this to render the workflow progress view, avoiding a separate API call to `/workflow/definitions/{id}` (which may have different permission requirements).
- **Permission gap for workflow definition read:** `WorkflowReader` requires `project:read` permission, but the 项目经理 role doesn't have it. The `workflow-status` endpoint was changed to use `CurrentUserWithAccess` (requires `system:access`) instead.

- [2026-06-01] NEVER try to run the backend locally or connect to PostgreSQL from the host machine. The development environment runs in Docker (`docker compose -p eai-docker`). After code changes, restart the container (`docker compose -p eai-docker restart gateway`). For database queries, use `docker exec <postgres-container> psql -U agentflow -d agentflow`.
- [2026-06-01] NEVER try to start dev services with `make dev` or kill chrome processes to test. Always assume the Docker environment is already running. Restart specific containers after code changes.
- [2026-06-01] NEVER try to decode Yjs CRDT binary (CollabVersion.snapshot / CollabDocument.yjs_doc) as UTF-8 text — it produces garbled output. Always use the `snapshot_text` column (markdown) for diff and summary operations.
- [2026-05-16] Never create a fresh SQLAlchemy engine per request in `get_db_context()` — it kills performance under even minimal load. Always reuse the shared engine pool.
- [2026-05-28] Do not try to import from `y-protocols/awareness` in frontend code — pnpm doesn't hoist it to top-level node_modules. Use `as any` cast for BlockNote's provider type instead.
- [2026-05-29] Do not add pre-compiled packages (like @blocknote/xl-ai) to `transpilePackages` in next.config.js — Turbopack's transpilePackages breaks subpath export resolution (e.g. `@scope/pkg/locales`), causing imports to resolve as `undefined` at runtime with no build-time error.
- [2026-06-01] ProseMirror `Step.from` and `StepMap.ranges` are not accessible in TypeScript — the types don't expose them. For human-written detection, use `tr.doc.forEach()` to mark all blocks on any local edit instead of trying to trace individual step positions.
- [2026-05-30] `@blocknote/xl-ai/server` subpath import is broken in Turbopack — `aiDocumentFormats` resolves to `undefined`. Do not import from this path in Next.js API routes. Instead, implement the AI chat endpoint manually using direct `fetch()` to the LLM API with SSE stream forwarding.
- [2026-05-30] `@ai-sdk/openai` v2 defaults to the Responses API (`/v1/responses`). Use `openai.chat()` method to use the Chat Completions API. But even `openai.chat()` converts `system` role to `developer` role which non-OpenAI providers (DeepSeek, Zhipu) don't support. For non-OpenAI providers, bypass the AI SDK entirely and call the API directly.
- [2026-05-30] Next.js external rewrites (with full URL destinations) in `afterFiles` array intercept requests before API route handlers. Use `fallback` array for catch-all API rewrites so that local Next.js API routes take priority.
- [2026-05-30] Docker nginx `proxy_pass http://frontend` (no port) defaults to port 80, but Next.js container listens on port 3000. Always use explicit port: `deer-flow-frontend:3000`.
- [2026-05-30] Do NOT emit `{ type: "tool-call" }` in SSE stream for AI SDK's `DefaultChatTransport` — it triggers `AI_TypeValidationError`. The valid chunk types are `tool-input-start`, `tool-input-delta`, `tool-input-available`, `tool-output-available`, etc.
- [2026-05-30] The `@blocknote/xl-ai` AIMenu does NOT render built-in accept/reject buttons. You MUST return accept/reject items from the `items` callback when `status === "user-reviewing"`.
- [2026-06-02] NEVER store a React state snapshot of a derived value (like selectedNode from a nodes array) — store only the ID and derive via useMemo. Storing a snapshot means edits via updateNodeData won't reflect because the component reads the stale copy. This was the root cause of WorkflowEditor property panel inputs being non-interactive.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->
- [2026-05-28] Chose BlockNote over Tiptap for the collab editor. BlockNote v0.51 provides native Yjs collaboration support (fragment + provider + user + showCursorLabels), Notion-style block editing, and built-in markdown import/export (blocksToMarkdownLossy, tryParseMarkdownToBlocks). The collaboration fragment key is "document-store" (not "thread").

- **Collab seeding flow:** When a document is first opened in the BlockNote collab editor, the Hocuspocus server's `onLoadDocument` loads from `collab_documents.yjs_doc`. If no entry exists, it sets `pendingMarkdown` in Yjs metadata (`_collabMeta`). The client's `BlockNoteEditor.tsx` reads this metadata, seeds from the markdown using `tryParseMarkdownToBlocks()`, then clears the flag. This ensures content is always seeded from the correct source (`ai_documents.content` or file_ref file content).
- **BlockNote `tryParseMarkdownToBlocks` limitations:** BlockNote's built-in markdown parser has limited support for complex markdown (tables, nested lists, mixed Chinese/English). For complex technical documents, the parsed blocks may differ from the original markdown structure.
- **Two-storage system pattern:** The collab system has two storage tables: `ai_documents.content` (original markdown) and `collab_documents.yjs_doc` (Yjs binary). They must be kept in sync through the seeding mechanism. Never use `blocksToMarkdownLossy()` to overwrite the original markdown from collab editor changes.
