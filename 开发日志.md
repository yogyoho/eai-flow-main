修复长报告流式输出时对话页面卡顿问题

     背景

     当 AI 流式输出长报告（数千 tokens）时,对话页面严重卡顿。根因是每个 SSE 事件都触发完整的 React
     重渲染,且每次重渲染都对完整的累积内容进行全量 markdown 解析和 DOM 扩展。

     瓶颈分析（按影响排序）

     ┌────────┬─────────────────────────────────┬───────────────────────────┬──────────────────────────────────────────┐
     │ 优先级 │              瓶颈               │           位置            │                   影响                   │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P0     │ useStream 未开启 throttle       │ hooks.ts:161              │ 每个 token 触发一次完整渲染,万 token =   │
     │        │                                 │                           │ 万次渲染                                 │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P0     │ Intl.Segmenter 每次创建新实例   │ rehype/index.ts:25        │ 每个文本节点都 new 一次,大量 GC 压力     │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P1     │ mergedThread 每次渲染产生新引用 │ hooks.ts:505-516          │ 所有 ThreadContext 消费者无条件重渲染    │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P1     │ getMessageGroups 未 memo        │ message-list.tsx:180      │ 每次渲染遍历所有消息做正则运算           │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P1     │ [...rehypePlugins, ...] spread  │ message-list-item.tsx:339 │ 每次渲染 MarkdownContent 收到新 prop     │
     │        │ 创建新数组                      │                           │                                          │
     └────────┴─────────────────────────────────┴───────────────────────────┴──────────────────────────────────────────┘

     修改方案

     修改 1 — 开启流节流（P0，一行改动）

     文件: frontend/src/core/threads/hooks.ts 第 161 行

     在 useStream 的 options 对象中添加 throttle: true:

     const thread = useStream<AgentThreadState>({
         client: getAPIClient(isMock),
         assistantId: "lead_agent",
         threadId: onStreamThreadId,
         reconnectOnMount: true,
         throttle: true,  // ← 新增
         fetchStateHistory: { limit: 1 },
         // ...
     });

     原理: SDK 的 StreamManager 在 throttle 为 truthy 时用 setTimeout(fn, 0) 批量合并同一事件循环内的多次 notify,将
     ~10000 次渲染削减为几十次。

     风险: 极低。这是 SDK 官方支持的选项（类型定义中 @default true）。

     ---
     修改 2 — 缓存 Intl.Segmenter 实例（P0，小改动）

     // 模块作用域（函数外）添加:
     let _segmenter: Intl.Segmenter | undefined;

     export function rehypeSplitWordsIntoSpans() {
       return (tree: Root) => {
         const segmenter = (_segmenter ??= new Intl.Segmenter("zh", { granularity: "word" }));
         // ... 将第 25 行的 new Intl.Segmenter(...) 替换为 segmenter

     原理: Intl.Segmenter 是无状态的（每次 .segment() 调用是纯函数）,可以安全复用。

     风险: 极低。

     ---
     修改 3 — memo mergedThread（P1，小改动）

     文件: frontend/src/core/threads/hooks.ts 第 505-516 行

     用 useMemo 包裹:

     const mergedMessages = useMemo(
         () => mergeMessages(history, thread.messages, optimisticMessages),
         [history, thread.messages, optimisticMessages],
     );

     const mergedThread = useMemo(
         () => ({ ...thread, messages: mergedMessages }) as typeof thread,
         [thread, mergedMessages],
     );

     原理: 避免每次渲染都创建新引用,防止 ThreadContext.Provider 的所有消费者无条件重渲染。

     风险: 低。标准 React 模式。
