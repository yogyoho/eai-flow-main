修复长报告流式输出时对话页面卡顿问题

     背景

     当 AI 流式输出长报告（数千 tokens）时,对话页面严重卡顿。根因是每个 SSE 事件都触发完整的 React
     重渲染,且每次重渲染都对完整的累积内容进行全量 markdown 解析和 DOM 扩展。

     瓶颈分析（按影响排序）

     │ P0     │ useStream 未开启 throttle       │ hooks.ts:161              │ 每个 token 触发一次完整渲染,万 token =   │
     │        │                                 │                           │ 万次渲染                                 │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P0     │ Intl.Segmenter 每次创建新实例   │ rehype/index.ts:25        │ 每个文本节点都 new 一次,大量 GC 压力     │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤
     │ P1     │ mergedThread 每次渲染产生新引用 │ hooks.ts:505-516          │ 所有 ThreadContext 消费者无条件重渲染    │
     ├────────┼─────────────────────────────────┼───────────────────────────┼──────────────────────────────────────────┤

     修改 1 — 开启流节流（P0，一行改动）

     文件: frontend/src/core/threads/hooks.ts 第 161 行

     在 useStream 的 options 对象中添加 throttle: true:

     const thread = useStream<AgentThreadState>({
         client: getAPIClient(isMock),
         assistantId: "lead_agent",
         threadId: onStreamThreadId,
         reconnectOnMount: true,
         throttle: true,  // ← 新增

docker/.env vs docker/.env.docker 区别

这两个文件分工明确，但容易混淆：

docker/.env — Docker Compose 变量替换文件

Docker Compose 在解析 docker-compose*.yaml 时自动读取同目录下的 .env 文件，用于替换 compose 文件中的 ${VAR} 占位符。

docker compose -f docker-compose-dev.yaml up
                ↓ 自动读取
           docker/.env
                ↓ 替换
    "${POSTGRES_EXT_PORT:-5432}"  →  "5432"

作用时机：docker compose 命令执行时，容器启动前。只影响 compose 文件中的 ${VAR} 变量替换和容器的端口映射。

docker/.env.docker — 容器内环境变量注入文件

通过 compose 文件中的 env_file 指令显式注入到容器内部：

# docker-compose-dev.yaml
gateway:
  env_file:
    - ../.env        # 项目根目录的 .env

或

# docker-compose.yaml (prod)
gateway:
  env_file:
    - ../.env

作用时机：容器启动后，作为容器内的环境变量。影响应用运行时的行为（如数据库连接字符串、API 密钥等）。

对比总结

│    维度    │       docker/.env        │   docker/.env.docker
├────────────┼──────────────────────────┼─────────────────────────┤
│ 读取方     │ Docker Compose 引擎      │ 容器内应用程序
├────────────┼──────────────────────────┼─────────────────────────┤
│ 作用       │ 替换 ${VAR} 定义端口映射 │ 注入运行时环境变量
├────────────┼──────────────────────────┼─────────────────────────┤

实际工作流

docker compose -p eai-docker -f docker-compose-dev.yaml up

  1. 读取 docker/.env
     ${POSTGRES_EXT_PORT} → 5432
     ${RAGFLOW_PORT}      → 9380
     ...

  2. 解析 compose 文件，替换所有 ${VAR}
     ports: "${POSTGRES_EXT_PORT}:5432" → "5432:5432"

  3. 创建容器，映射端口
     host:5432 → container:5432

  4. 容器启动后，通过 env_file 注入 .env.docker
     容器内可访问 $DEER_FLOW_HOME, $BETTER_AUTH_SECRET 等

为什么端口改 docker/.env 就够了                                                                            
因为端口映射只在 compose 文件解析阶段需要，改 .env 后 ${POSTGRES_EXT_PORT} 自动变为 5432，compose 就会映射 5432:5432。容器内的应用程序不关心宿主机的映射端口，它们只通过 ext:5432）互相访问。

   分析：Docker 容器重建后对话历史为空                                                                                                                    ↑

   背景                                                                                                                                                   ↑

   用户在开发环境中多次删除并重建 Docker 容器后，发现对话页面的历史列表可以看到历史会话（标题、时间），但点击进去后对话内容为空。                         ↑

   根因分析                                                                                                                                               ↑

   问题出在两套数据库存储了不同维度的数据，且关键数据库放在了 Docker 容器的临时目录中。                                                                   ↑

   数据存储架构                                                                                                                                           ↑

   ┌────────────────┬──────────────────────────────────────────┬──────────────────────────────────────┬───────────────────────────────┐                   ↑
   │     数据库     │                   路径                   │               存储内容               │          是否持久化           │
   ├────────────────┼──────────────────────────────────────────┼──────────────────────────────────────┼───────────────────────────────┤                   ↑
   │ deerflow.db    │ /app/backend/.deer-flow/data/deerflow.db │ 线程元数据（标题、时间、user_id）    │ ✅ 是（bind mount 到宿主机）  │
   ├────────────────┼──────────────────────────────────────────┼──────────────────────────────────────┼───────────────────────────────┤                   ↑
   │ checkpoints.db │ /tmp/deerflow/checkpoints.db             │ LangGraph 检查点状态（实际对话消息） │ ❌ 否（容器内 /tmp 是临时的） │
   └────────────────┴──────────────────────────────────────────┴──────────────────────────────────────┴───────────────────────────────┘                   ↑

   证据                                                                                                                                                   ↑

   1. config.yaml 第 196 行：                                                                                                                             ↑
   checkpointer:
     type: sqlite                                                                                                                                         ↑
     connection_string: /tmp/deerflow/checkpoints.db  # ← 绝对路径 /tmp 是临时目录
   2. 容器内实际状态：                                                                                                                                    ↑
     - /tmp/deerflow/checkpoints.db = 40KB（今天容器重建后新生成的，几乎为空）
     - /app/backend/.deer-flow/checkpoints.db = 52MB（5月16日，包含所有历史数据）                                                                         ↑
   3. 路径解析逻辑（paths.py:345-354）：resolve_path() 对绝对路径直接返回，不进行任何转换，所以 /tmp/deerflow/checkpoints.db 就是容器内的 /tmp 目录。
   4. 数据流：                                                                                                                                            ↑
     - 前端左侧对话列表 → 查询 deerflow.db（线程元数据）→ 有数据 → 显示历史列表 ✅
     - 前端点击进入对话 → 查询 checkpoints.db（消息内容）→ 新库无数据 → 显示空白 ❌                                                                       ↑

   为什么会发生                                                                                                                                           ↑

   - /tmp 是 Docker 容器的临时文件系统，容器删除后所有数据丢失                                                                                            ↑
   - docker compose down && docker compose up -d 会删除旧容器并创建新容器
   - 新的 /tmp/deerflow/checkpoints.db 是空的，LangGraph 无法恢复历史消息                                                                                 ↑

   修复方案                                                                                                                                               ↑

   修改 config.yaml，将 checkpointer 路径从 /tmp/deerflow/checkpoints.db 改为基于 DEER_FLOW_HOME 的相对路径：                                             ↑

   # 修改前                                                                                                                                               ↑
   checkpointer:
     type: sqlite                                                                                                                                         ↑
     connection_string: /tmp/deerflow/checkpoints.db
                                                                                                                                                          ↑
   # 修改后                                                                                                                                               ↑
   checkpointer:
     type: sqlite
     connection_string: .deer-flow/checkpoints.db

   使用相对路径 .deer-flow/checkpoints.db 后，resolve_path() 会将其解析为 {DEER_FLOW_HOME}/.deer-flow/checkpoints.db，即 /app/backend/.deer-flow/checkpoints.db，这个路径通过 bind mount 映射到宿主机 D:\eai\eai-flow-main\backend\.deer-flow\checkpoints.db。

   为什么这样修复

DEER_FLOW_HOME=/app/backend/.deer-flow（在 docker-compose-dev.yaml 中设置）
   - /app/backend/ 通过 bind mount 映射到宿主机目录
   - 宿主机上已有 52MB 的历史 checkpoints.db（虽然数据是5月的，但至少比全空好）
   - 以后新产生的对话数据也会持久化到宿主机

 注意事项

   1. 修改后需要重启 gateway 容器：docker compose -p eai-docker restart gateway
 宿主机上现有的 D:\eai\eai-flow-main\backend\.deer-flow\checkpoints.db（52MB）会被使用，其中包含5月16日之前的历史数据
   3. 5月16日至今（使用 /tmp 路径期间）产生的对话数据已经丢失，无法恢复

   验证方法

 修改 config.yaml 后重启 gateway
   2. 检查日志确认 checkpointer 使用了新路径：
   docker compose -p eai-docker logs gateway | grep -i "SqliteSaver\|checkpointer"
   3. 在浏览器中打开对话历史，点击旧对话查看是否能加载出历史消息
 4. 发起新对话验证新消息能正常持久化
