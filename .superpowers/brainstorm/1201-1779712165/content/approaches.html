协作智能写作 — 3种架构方案

基于 DeerFlow 现有系统，如何扩展出工程报告协作写作能力

A

方案A: Report Orchestration Layer（报告编排层）

在现有 Agent 系统之上新增一个报告编排层

┌─ ReportOrchestrator (新增)
│
├─ ReportProject (报告项目)
│ ├─ 章节列表 + 状态机 (draft→collecting→writing→review→done)
│ ├─ 章节锁 (owner + lock_type)
│ └─ 实时预览 (SSE 广播)
│
├─ 3个专用 Subagent
│ ├─ data-collector: RAG检索 + 数据提取
│ ├─ report-writer: 按模板章节编写
│ └─ compliance-checker: 合规规则校验
│
└─ 编排引擎
├─ 按章节拆分 → 分配 Agent → 收集结果
└─ 复用 knowledge_factory 的模板/合规引擎
────────────────────────────────
现有系统（不修改）
├─ Lead Agent + SubagentExecutor
├─ Knowledge Factory (模板/合规)
├─ TiptapEditor (文档编辑)
└─ ThreadState + StreamBridge

优势

  • 不修改现有 harness 核心，纯 App 层扩展
  • 复用 knowledge_factory 模板+合规引擎
  • 编排逻辑清晰，易于理解和调试

劣势

  • Subagent 15分钟超时限制（长报告可能超时）
  • 每个章节独立 Agent 无共享上下文
B

方案B: Agent Workflow Pipeline（Agent 工作流）

用 LangGraph StateGraph 构建报告生成流水线

ReportWorkflowGraph (LangGraph StateGraph)
START → [章节拆分] → [并行: 每章节]
→ 数据收集 → AI编写 → 合规校验 → 人工审核(挂起等待)
→ [合并章节] → [全文一致性检查] → END
ReportState (LangGraph 状态)
sections: dict[章节ID, SectionState]
shared_context: 全局上下文(术语表、参数)
status: draft→generating→reviewing→approved

优势

  • 原生 LangGraph 流水线，可持久化、可恢复
  • 章节间共享状态（术语、参数自动对齐）
  • 天然支持并行处理多章节
  • 人工审核可作为图中的 interrupt 节点

劣势

  • 需要新建 LangGraph 图，复杂度较高
  • 与现有 Lead Agent 是两套运行时
  • 实时协作需要额外的 SSE 层
C

方案C: Hybrid（混合方案）

编排层 + 轻量 LangGraph 流水线，取两者之长

┌─ ReportOrchestrator (App层)
│ ├─ 项目管理 CRUD
│ ├─ 章节锁 + SSE 实时预览
│ └─ 协作 API (Gateway Router)
│
├─ SectionPipeline (轻量 LangGraph)
│ 每个章节一个轻量 StateGraph:
│ collect → write → validate → review
│ (带 interrupt 支持人工挂起)
│
├─ Agent 注册 (deerflow/subagents/)
│ ├─ data-collector (专用)
│ ├─ report-writer (专用)
│ └─ compliance-checker (复用引擎)
│
└─ 复用层
├─ knowledge_factory 模板+合规
├─ TiptapEditor AI内联操作
└─ StreamBridge SSE

优势

  • 项目管理由 App 层处理（简单 CRUD）
  • 章节流水线用 LangGraph（可恢复、可中断）
  • 3个专用 Agent 各司其职
  • 与现有架构完美契合

劣势

  • 需要同时在 App 和 Harness 层开发
  • 章节间共享上下文需要额外设计
