设计 Section 3: 工作流执行引擎

Temporal DynamicGraphWorkflow 解释器 + 三大核心流程

引擎核心：DAG → Temporal Workflow 映射
设计思路：DAG JSON 存在应用库，Temporal 只负责执行。 DynamicGraphWorkflow 是一个通用工作流，读取 DAG JSON 拓扑排序后逐步执行节点。

DAG 节点类型 → Temporal 行为映射

节点类型 Temporal 行为 信号/等待机制
phase 执行阶段 Activity（创建子项目、分配团队、AI 撰写） wait_condition 等待 phase:complete 信号
review 创建审核分配（phase_reviews），等待所有人完成 wait_condition 等待 review:complete 信号
rejected → 沿边回退到目标节点
condition 评估表达式，选择分支 无等待，立即路由
ai_generate 调用 DeerFlow Agent 撰写章节 wait_condition 等待 ai:complete 信号
merge 汇聚并行分支的输出，作为下一阶段的上下文 wait_condition 等待所有上游分支完成

三大核心流程

Flow 1: 多阶段项目（串行 + 并行 + 上下文传递）
阶段A
→ Activity: init_phase(team-a, chapters-a)
→ wait signal: phase:complete
↓ 自动传递上下文 ↓
阶段B
asyncio.gather()
阶段C
asyncio.gather()
↓ merge 节点汇聚 ↓
阶段D
→ 上下文包含 B+C 全部输出
上下文传递机制
每个阶段完成时，Activity 将阶段产物（章节内容、关键结论、数据摘要）写入 phase_context JSON 字段。

下游阶段启动时，Temporal 将上游 context 作为输入传给 Activity。

并行汇聚时，merge 节点合并所有上游 context 为数组。
Flow 2: AI 内容溯源（段落级）
溯源采集流程：
  1. Agent 生成内容时：report-write Skill 要求 AI 在输出中标注 [source:type:id] 标记
  2. 写入章节时：MCP write_chapter Activity 解析标记，提取到 content_sources 表
  3. 编辑器展示时：BlockNote 自定义扩展渲染溯源标注（类似 Wikipedia 脚注）
  4. 人工编辑时：自动标记为 human_written 类型
// AI 生成时的标注格式
该区域 SO₂ 日均浓度为
0.045mg/m³[1]
，低于国家标准限值
0.15mg/m³[2]
// 溯源脚注（自动提取到 content_sources）
[1] 知识库「监测数据库」/ 文档「2024年度监测报告.pdf」p.23
[2] 法规库 / GB 3095-2012 表2
Flow 3: 多人分章节/维度混合审核
模式A: 按章节
review_type=chapter
张三→第1-3章
李四→第4-6章
模式B: 按维度
review_type=dimension
张三→技术维度
李四→合规维度
模式C: 混合
同一个 review 节点
既有章节分配
也有维度分工
Temporal 执行逻辑：
  1. review 节点 Activity 创建 N 条 phase_records
  2. Temporal wait_condition 等待所有 records 完成
  3. 任一 rejected → signal 回退到配置的目标阶段
  4. 全部 approved → 工作流推进到下游节点
phase_reviews 表查询示例：
-- 按章节审核
WHERE phase_node='review-1'
AND project_id='...'
AND review_type='chapter'
AND chapter_id IS NOT NULL
-- 按维度审核
WHERE phase_node='review-1'
AND project_id='...'
AND review_type='dimension'
AND dimension='compliance'
