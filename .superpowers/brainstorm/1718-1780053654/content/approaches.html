三种架构方案对比

核心决策：自研 vs 集成 vs 混合 — 点击选择你倾向的方案

A

集成 Temporal.io 工作流引擎

思路：用 Temporal 作为工作流运行时，前端用 React Flow 构建可视化编辑器，后端通过 Temporal SDK 定义工作流。报告项目是 Temporal 工作流的一个实例。

优势

  • 生产级工作流引擎，可靠的状态持久化
  • 原生支持串行/并行/条件/子流程/错误处理
  • 工作流版本控制和演进能力
  • 成熟的可视化和监控工具
  • Python + TypeScript SDK 都有

代价

  • 新增基础设施依赖（Temporal Server）
  • 学习曲线：Temporal 的概念模型需要团队掌握
  • 工作流代码是确定性约束的，开发模式不同
  • Docker 部署增加 3 个容器

开发量：前端 React Flow 编辑器 (~2周) + Temporal 集成 (~1周) + 业务逻辑 (~1周) = ~4周

B

自研轻量工作流引擎 + React Flow

思路：后端用 PostgreSQL 存储工作流定义（DAG JSON），自研执行引擎处理节点状态转换。前端用 React Flow 做可视化编辑器。不引入外部工作流引擎。

优势

  • 零外部依赖，不需要新增基础设施
  • 完全控制数据模型，与现有 ReportProject 无缝融合
  • 渐进式扩展：先做 DAG，后加条件分支
  • 对团队来说概念简单，就是状态机 + DAG

代价

  • 条件分支、错误恢复、并发控制需要自己实现
  • 工作流引擎的边界情况多，测试量大
  • 没有现成的监控/调试工具
  • 如果需求继续复杂化，可能最终还是要迁移到 Temporal

开发量：React Flow 编辑器 (~2周) + 引擎核心 (~2周) + 业务集成 (~1周) = ~5周

C

渐进式混合：先自研核心 + 预留 Temporal 接口

思路：Phase 1 用自研 DAG 引擎 + React Flow 跑通核心场景（串行/并行/审核/子项目）。设计好 WorkflowEngine 抽象接口，Phase 2 可平滑迁移到 Temporal。

优势

  • Phase 1 快速交付（无新基础设施）
  • 抽象接口确保未来可替换引擎
  • 先用真实业务验证需求，再决定是否上 Temporal
  • 降低一次性投入风险

代价

  • 自研引擎可能需要在迁移时重写
  • 抽象层增加一点复杂度
  • 如果业务爆发式增长，迁移窗口不好找

开发量：Phase 1 ~3周（自研 DAG + React Flow）→ Phase 2 可选迁移 ~2周

三个方案的决策矩阵

维度 A: Temporal B: 纯自研 C: 渐进混合
基础设施复杂度 高 (+3容器) 零 零 → 可选
条件分支/循环 原生支持 需自研 Phase 1 无 → Phase 2 有
与现有模型融合 需要适配层 无缝 无缝
首版交付速度 ~4周 ~5周 ~3周
长期可维护性 强（社区+文档） 弱（自研代码） 中（可迁移）
溯源 + 审核集成 通过 Activity API 直接数据库操作 直接数据库操作
