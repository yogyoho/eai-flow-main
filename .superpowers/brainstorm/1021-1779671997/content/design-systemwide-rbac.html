📐 统一 RBAC：两层权限模型

┌─────────────────────────────────────────────────────────┐
│                 统一权限架构 (Unified RBAC)                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  第 1 层：系统级权限 (System Permissions)                  │
│  ─────────────────────────────────────                   │
│  存储位置：Role.permissions[] 数组                        │
│  格式：     "resource:action"                            │
│  示例：     "kb:read", "user:create", "project:create"   │
│  检查方式： require_permission("kb:read")  ← 已有         │
│  适用范围： 用户能否访问某类功能模块                        │
│                                                         │
│  第 2 层：资源级权限 (Resource Permissions)                │
│  ─────────────────────────────────────                   │
│  存储位置：PERMISSION_MATRIX + 成员角色                    │
│  格式：     "resource:action" (同上)                      │
│  示例：     "project:member:add", "project:outline:edit"  │
│  检查方式： require_resource_permission(pid, action)  ←新  │
│  适用范围： 用户在特定项目/资源内能做什么                   │
│                                                         │
│  统一接口：can(user, action, resource?) → bool            │
│  ─────────────────────────────────────                   │
│  无 resource → 查系统级 Role.permissions                 │
│  有 resource → 查资源级 成员角色 + PERMISSION_MATRIX       │
│                                                         │
└─────────────────────────────────────────────────────────┘

🔄 统一权限检查函数

# backend/app/extensions/auth/permissions.py — 统一权限服务

class PermissionService:
    """统一的权限检查入口，覆盖系统级 + 资源级"""

    @staticmethod
    async def can(
        user_id: UUID,
        action: str,
        resource_type: str | None = None,  # "project", "kb", etc.
        resource_id: UUID | None = None,
        db: AsyncSession = ...,
    ) -> bool:
        # 1. 获取用户系统角色
        user = await db.get(User, user_id)
        role = await db.get(Role, user.role_id)

        # 2. 系统级权限：通配符或直接匹配
        if "*" in role.permissions or action in role.permissions:
            return True

        # 3. 资源级权限：查成员角色 + 权限矩阵
        if resource_type and resource_id:
            member_role = await _get_resource_member_role(
                db, resource_type, resource_id, user_id
            )
            if member_role:
                return check_resource_permission(member_role, action)

        return False

# 使用 — 与现有 require_permission 兼容，同时支持资源级：
# 系统级（已有，不变）：
@router.get("/knowledge-bases")
async def list_kb(user = Depends(require_permission("kb:read"))):
    ...

# 资源级（新增）：
@router.post("/projects/{pid}/members")
async def add_member(
    pid: UUID,
    user = Depends(require_resource_permission("project", pid, "member:add")),
):
    ...

✅ 升级策略：渐进式，不破坏现有代码

  • Phase 1（本次）：项目管理模块接入资源级权限，新建 PermissionService
  • Phase 2：知识工厂、法规库、文档管理升级为细粒度系统级权限
  • Phase 3：知识库、文件夹等增加资源级权限（按需）
  • 不变：现有 require_permission("kb:read") 调用全部兼容，无需修改
